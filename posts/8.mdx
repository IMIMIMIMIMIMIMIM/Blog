프론트엔드 개발에서 SSR(Server-Side Rendering)과 CSR(Client-Side Rendering)은 대표적인 렌더링 방식이다  
일반적으로 SSR이 무조건 더 좋다고 생각하기 쉽지만 실제로는 서비스의 특성에 따라 장단점이 뚜렷하다  
아래는 두 방식의 특징을 정리한 내용이다

## SSR(Server-Side Rendering)

### 장점

- 초기 렌더링 속도가 빠르다: 서버에서 완성된 HTML을 내려주기 때문에 첫 화면 표시 시간이 짧다
- SEO에 유리하다: 검색엔진 크롤러가 완성된 HTML을 바로 인식할 수 있다
- 소셜 미리보기에 안정적이다: 메타 태그가 서버에서 바로 적용되어 공유 시 올바른 정보가 표시된다
- 저사양 기기에서도 유리하다: 클라이언트에서의 연산이 줄어든다

### 단점

- 서버 부하가 증가한다: 모든 요청에 대해 HTML을 매번 생성해야 한다
- 캐싱 효율이 낮다: 페이지가 동적으로 생성되므로 정적 리소스 캐싱에 비해 부담이 크다
- 구현 복잡도가 높다: 서버 환경 설정과 클라이언트 연동 로직이 필요하다
- 사용자 인터랙션이 많은 경우 비효율적이다

## CSR(Client-Side Rendering)

### 장점

- 초기 로딩 후 사용자 경험이 우수하다: 페이지 전환이 빠르고 부드럽다
- 서버 부하가 적다: 서버는 정적 자바스크립트 파일만 제공하면 된다
- SPA 구조에 적합하다: React나 Vue 같은 프레임워크와 결합 시 개발이 단순하다
- 캐싱에 유리하다: 정적 리소스를 효과적으로 캐싱할 수 있다

### 단점

- 초기 로딩 속도가 느리다: 클라이언트에서 자바스크립트를 해석하고 렌더링해야 한다
- SEO에 불리하다: 크롤러가 자바스크립트 실행을 완벽히 처리하지 못할 수 있다
- 소셜 미리보기에 제약이 있다: 메타 태그가 동적으로 변경될 경우 반영되지 않을 수 있다
- 저사양 기기에서는 성능 저하가 발생한다

## 결론

SSR은 초기 표시 속도와 SEO가 중요한 서비스에서 적합하다  
반대로 CSR은 로그인 기반 서비스나 대시보드처럼 사용자 인터랙션이 많고 SEO가 필요 없는 환경에서 유리하다  
따라서 SSR이 CSR보다 반드시 좋은 것은 아니며 서비스 특성에 맞추어 선택해야 한다  
최근에는 Next.js 같은 프레임워크를 통해 SSR과 CSR을 혼합 적용하는 하이브리드 접근 방식이 일반적이다
